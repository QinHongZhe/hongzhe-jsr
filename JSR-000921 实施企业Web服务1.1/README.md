2.1请描述拟议的规范：
该规范定义了用于在Java中实现Web服务的编程模型和体系结构。该规范将基于JSR 67、93和101的工作。该规范还将基于J2EE技术​​的JSR，包括J2EE本身，Servlet和JSP。该JSR的目的不是服从J2EE JSR的角色，也不是为了定义平台。我们也不以为该JSR的输出将成为J2EE的一部分为前提。在J2EE JSR流程中将决定完全或部分选择要包含在J2EE中的JSR输出。

JSR 101专注于XML RPC和Java语言，包括表示Java中基于XML的接口定义，基于XML接口定义语言（例如SOAP）的Java定义和编组。JSR 67为XML消息传递提供了类似的功能。

JSR 93定义了到XML注册中心的Java接口，例如JNDI，ebXML和UDDI。这些接口提供了一种机制，通过该机制，客户端应用程序可以找到Web服务，然后Web服务（和服务器）发布其接口。

此JSR的目标是为Web服务提供一个基于JSR 67、93和101以及面向单个Web服务标准的未来JSR的Web服务编程模型和运行时模型，类似于EJB规范对RMI（RMI-IIOP）和JNDI所做的工作。这仅是一个类比，并且此JSR将建立在EJB规范的基础上，但不会扩展EJB规范。

具体来说，我们将把这个JSR集中在：

用于实现Web服务的编程模型。这可能包括定义新的服务器端基类和框架，指定新的API，定义JSP，Servlet或现有EJB类型（如MessageBean）的新的具体子类。任何扩展都将类似于HttpServlet扩展通用Servlet或Enterprise MediaBeans定义EntityBeans子类型所采用的方法。
用于使用来自Java的Web服务的客户端编程模型。这将类似于EJB定义的客户端编程模型。这将解释如何串联使用JSR 67、93和101中定义的API。再次，目的是使引入的新概念最小化。
该规范将说明现有概念（如EJB事务模型，Servlet，EJB的安全性或HttpSession State）如何在Web服务的使用和实现中实现。
定义如何扩展基本的Servlet / HTTP模型，使其包括通过FTP，电子邮件等分发Web服务。同样，理想情况下，这还是引用了现有的JSR / Java标准，并着重于支持Web服务所需的编程模型和部分。
定义用于在J2EE之上开发和部署Web服务的具体模型。

该JSR将提供有关编程模型，API和运行时服务模型的文档。它将为任何符合J2EE的应用服务器提供参考实现，并具有用于互操作性和合规性的开源测试用例。该规范将依赖于现有的J2EE应用程序包装。

此JSR可能回答的一些示例问题是：

Web服务的生命周期模型是什么？
是否有无状态和有状态的Web服务？您如何实现有状态EJB？
Web服务有家吗？
是否有用于Web服务的EntityBean模型？
在ebXML或SOAP上流到EJB服务的映射服务上下文是什么？
Web服务是否需要新的服务器端类？现有的J2EE服务器端API（例如EJB和JMS）可在多大程度上用于调度Web服务调用？子类适合于特定于Web服务的EJB，还是应该将Web服务信息放入EJB部署描述符中，还是都需要两种技术？
是否需要新的Web服务客户端类？使用现有的J2EE客户端编程模型（例如EJB和JMS）可以在多大程度上调用Web服务？
2.2目标Java平台是什么？（即台式机，服务器，个人，嵌入式，卡等）
该规范针对J2EE平台。

2.3提议的规范将满足Java社区的哪些需求？
在过去的几年中，在许多供应商的帮助下，J2EE已成为开发基于Web的业务应用程序的首选平台。在主要标准机构（例如，万维网联盟（W3C），ebXML集团和UDDI组织）的协助下，正在开发标准，以动态发布，查找和绑定到Web上的业务应用程序。这些业务应用程序可以用Java或任何其他编程语言编写，但是只要它们遵循适当的标准，就可以作为Web服务参与。对于Java社区来说，非常重要的一点是Java应用程序程序员拥有一个通用的模型来在J2EE下编写和运行Web服务。有一个一致的Java模型来访问和使用公共Web服务协议定义的接口和服务，这一点很重要。这包括当今存在的那些（例如SOAP RPC，SOAP消息传递和WSDL）以及那些正在诸如安全性，信任和系统管理之类的领域中正在开发的事物。

该规范不会损害J2EE JSR的整体协调任务。该规范旨在定义程序员使用的API，程序员扩展的基本类型以及Web服务接口定义语言和服务（例如，安全性，SOAP附件）的通用运行时映射。

2.4现有规范为何不能满足这种需求？
已经开放了用于定义部件Web服务的API的规范，例如JSR 67（用于XML消息传递的Java API），JSR 93（用于XML注册表的Java API）和JSR 101（用于基于XML的RPC的Java API）。就像Servlet将一组概念（如cookie和HttpSession）捆绑在一起，以及EJB通过编程模型和运行时模型将RMI，JTA / JTS，JDBC等捆绑在一起一样，我们认为该JSR在实现和使用网络服务。
